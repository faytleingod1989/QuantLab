## 1. 产品概述

QuantLab 是一套面向个人和小型团队的 A 股量化策略研究与历史回测平台。产品仅覆盖上海证券交易所、深圳证券交易所和北京证券交易所上市的人民币普通股票（A 股）。用户可以导入标准化行情与证券基础数据，编写或配置交易策略，设置资金、交易成本和 A 股交易约束，运行历史回测，并通过净值曲线、交易记录及绩效指标评估策略的收益、风险和稳定性。

MVP 聚焦“A 股策略从想法到可信回测结果”的完整闭环，不包含自动实盘交易。平台优先保证复权、交易日历、T+1、涨跌停、停牌、ST 状态及交易费用等 A 股时间语义的正确性，而非追求复杂的高频撮合或全市场数据服务。

## 2. 背景与问题

量化策略开发通常面临以下问题：

- 策略代码、行情处理、回测逻辑和结果分析混杂，维护成本高。
- 简易回测容易出现未来函数、前视偏差、幸存者偏差等问题。
- 手续费、滑点、停牌、涨跌停等交易约束未被准确模拟，结果失真。
- 不同策略缺少统一的结果指标和对比方式。
- 回测过程难以复现，参数、数据版本和代码版本无法追踪。

本产品通过统一数据格式、标准策略接口、事件驱动回测引擎和可追溯的回测报告，降低策略验证门槛并提高研究可信度。

## 3. 产品目标

### 3.1 业务目标

- 让具备基础 Python 能力的用户在 30 分钟内完成首个策略回测。
- 建立可扩展的策略与回测底座，为后续模拟交易和实盘交易预留接口。
- 形成统一、可复现、可比较的策略评估流程。

### 3.2 用户目标

- 快速验证交易想法是否具有历史统计优势。
- 清晰了解策略收益来源、风险水平和交易行为。
- 通过参数实验和分阶段验证识别过拟合策略。
- 保存并复现任意一次回测。

### 3.3 MVP 成功指标

- 首次成功回测完成率不低于 80%。
- 日线级、单标的、10 年数据回测时间不超过 5 秒（普通个人电脑）。
- 相同输入条件下，回测结果必须完全一致。
- 核心指标与基准测试样例的计算误差不高于 0.1%。
- 严重级回测正确性缺陷为 0。

## 4. 产品范围

### 4.1 MVP 范围

- CSV 历史行情导入与数据校验。
- 日线级单标的及多标的回测。
- 覆盖沪市、深市和北交所 A 股现货；不同交易所、板块规则独立配置。
- A 股证券主数据、交易日历、上市/退市日期、ST 状态和复权因子管理。
- Python 代码型策略接口。
- 买入、卖出、持有、目标仓位等基础信号。
- 初始资金、仓位、手续费、印花税和滑点配置。
- 市价单及下一交易周期成交模拟。
- A 股约束：T+1 卖出限制、最小买入单位、涨跌停、停牌及退市状态处理。
- 基础风控：最大仓位、止盈、止损、不可交易证券过滤。
- 回测结果、交易流水、持仓流水与日志。
- 收益和风险指标、净值曲线、回撤曲线及月度收益。
- 回测配置保存、结果导出和历史任务查看。
- 买入并持有基准对比。

### 4.2 暂不纳入 MVP

- 券商接入及自动实盘交易。
- Tick 级、逐笔级和高频回测。
- 港股、美股、基金、可转债、期货、期权及其他非 A 股品种。
- 期权定价、复杂衍生品和保证金清算。
- 分布式计算与大规模参数集群优化。
- 用户付费、社交、策略商城和跟单。
- 平台内置商业行情采购与授权管理。

## 5. 用户角色与场景

### 5.1 个人量化研究者

具备基础 Python 或数据分析能力，希望快速验证均线、动量、轮动等策略，并理解策略风险。

### 5.2 策略开发者

关注回测引擎的准确性、扩展性和可复现性，需要自定义指标、信号、仓位及风控逻辑。

### 5.3 投研负责人（后续版本）

需要查看多策略表现、统一评价标准并管理研究结果，但不直接编写代码。

### 5.4 核心用户故事

- 作为研究者，我希望导入历史行情并查看质量报告，以确保数据可用于回测。
- 作为策略开发者，我希望通过统一接口编写策略，而无需重复实现账户和撮合逻辑。
- 作为研究者，我希望设置手续费、滑点和成交规则，以获得更接近真实交易的结果。
- 作为研究者，我希望查看收益、回撤、交易明细及基准对比，以判断策略是否值得继续研究。
- 作为研究者，我希望保存每次回测的参数和数据指纹，以便未来复现结果。

## 6. 核心业务流程

1. 创建项目或选择已有项目。
2. 导入行情数据并完成字段映射与质量校验。
3. 新建策略，选择模板或编写 Python 策略代码。
4. 配置回测区间、初始资金、标的池、交易成本、成交规则及风控参数。
5. 系统执行预检查，阻止缺失数据或无效配置进入回测。
6. 运行回测并展示进度、状态和运行日志。
7. 生成回测报告，展示绩效、净值、回撤、持仓和交易明细。
8. 保存本次配置与结果，支持导出或与其他回测对比。

## 8. 页面与信息架构

MVP 建议采用本地 Web 应用形式，包含以下页面：

- 工作台：A 股项目入口、最近策略、最近回测、主要指数概览和快捷操作。
- 数据中心：数据集列表、导入、质量报告和预览。
- 策略中心：策略列表、代码编辑器、参数和版本信息。

## 9. 核心规则与指标口径

### 9.1 默认交易规则

- T 日收盘数据产生信号，默认在下一交易日开盘撮合，严禁使用下一交易日数据生成信号。
- 普通 A 股买入委托数量按当日适用的交易单位取整；卖出零股、北交所及特殊业务规则由规则版本定义，不在代码中写死。
- 当日买入形成冻结可售数量，默认下一交易日才可卖出；系统分别记录持仓数量与可用数量。
- 买入成本包含成交金额、佣金、过户费和滑点；卖出成本另包含按当日政策适用的印花税。
- 资金不足时，系统按配置拒绝订单或缩减可买数量，默认拒绝。
- 不允许裸卖空；融资融券不属于 MVP。
- 缺少成交价格、标的停牌或处于不可交易状态时不成交，订单状态保留并按配置处理。
- 涨跌停不等于必然无法成交。MVP 默认采用保守模型，根据订单方向、当日价格边界及市场流动性代理条件判断成交，并允许切换模型。
- 证券上市前不得进入股票池；退市、风险警示和长期停牌必须在报告中单独披露。
- 策略计算可使用复权价格，撮合、现金和订单金额必须使用当日真实未复权价格。

### 9.2 核心指标

- 总收益率：期末净值相对初始净值的变化比例。
- 年化收益率：基于实际回测天数进行复合年化。
- 年化波动率：日收益率标准差乘以交易日年化因子。
- 最大回撤：历史净值相对此前峰值的最大跌幅。
- 夏普比率：超额收益的年化均值除以年化波动率；无风险利率可配置。
- 胜率：盈利的已平仓交易数占全部已平仓交易数的比例。
- 盈亏比：平均盈利金额除以平均亏损金额的绝对值。

所有指标必须在产品内展示计算口径、年化因子和无风险利率，避免同名指标口径不一致。

## 10. 非功能需求

### 10.1 正确性

- 账户、订单、成交和指标模块必须具备单元测试。
- 使用人工可计算的小型行情样例进行端到端金标准测试。
- 禁止未来数据访问，关键时间语义必须写入技术规范。
- 金额计算应明确精度与舍入方式。

### 10.2 性能

- 单标的 10 年日线数据应在 5 秒内完成回测。
- 100 个标的、5 年日线数据应在 60 秒内完成基础策略回测。
- 任务运行期间界面保持可用，并提供进度或运行状态。

### 10.3 稳定性与可维护性

- 数据、策略、回测引擎和报告模块解耦。
- 回测失败不得破坏已有数据和历史结果。
- 日志需包含任务 ID、策略 ID、数据版本和异常上下文。
- 核心接口和配置格式需要版本号，升级时提供兼容策略。

### 10.4 安全性

- 用户策略代码视为不可信代码；正式发布前应采用隔离进程运行并限制文件、网络和系统访问。
- 敏感配置不得写入策略文件或日志。
- 导入文件需限制类型、大小和解析资源，防止恶意文件导致服务不可用。

### 10.5 易用性

- 首次启动提供示例数据、示例策略和引导入口。
- 所有失败操作给出原因及可执行的修复建议。
- 核心指标、成本参数和交易术语提供就地解释。

## 12. MVP 验收标准

MVP 同时满足以下条件方可发布：

1. 用户可以导入符合模板的 CSV，并获得数据质量结果。
2. 用户可以运行内置双均线策略，也可以新增自定义 Python 策略。
3. 回测支持初始资金、佣金、最低佣金、印花税、过户费、滑点和历史规则版本。
4. 信号默认在下一交易周期成交，策略无法访问未来数据。
5. 系统正确执行 T+1、交易单位、停牌、涨跌停、上市及退市约束。
6. 系统能正确生成订单、成交、持仓、可用持仓、现金和每日净值流水。
7. 报告至少包含核心收益风险指标、净值曲线、回撤曲线和交易明细。
8. 任意回测均保存策略版本、参数、数据指纹、规则版本和引擎版本。
9. A 股金标准测试全部通过，无 P0/P1 正确性缺陷。
10. 示例用户从首次启动到完成首个回测不超过 10 分钟。

## 13. 里程碑建议

| 阶段 | 周期建议 | 交付内容 |
|---|---:|---|
| M0 技术验证 | 1 周 | 数据模型、策略接口、最小撮合与金标准样例 |
| M1 回测内核 | 2 周 | 账户、订单、成交、成本、净值和指标模块 |
| M2 可用产品 | 2 周 | 数据导入、策略编辑、任务管理和回测报告 |
| M3 质量加固 | 1 周 | 边界测试、性能优化、文档、示例和发布包 |

## 14. 风险与应对

| 风险 | 影响 | 应对方案 |
|---|---|---|
| 未来函数或时间语义错误 | 回测结果失真 | 强制数据窗口、下一周期成交、金标准测试 |
| 行情质量不一致 | 指标和成交异常 | 数据校验、复权标识、数据指纹和异常报告 |
| A 股历史规则变化 | 使用现行规则回测历史导致结果失真 | 规则按交易所、板块和生效日期版本化 |
| 复权口径混用 | 信号、成交和收益计算失真 | 研究价与成交价分离，保留复权因子和原始价格 |
| 涨跌停成交假设过度乐观 | 策略收益被高估 | 默认保守成交模型，并在报告披露未成交订单 |
| 策略代码失控 | 安全或稳定性问题 | 隔离进程、资源限制、超时与日志脱敏 |
| 参数过拟合 | 策略缺乏未来表现 | 后续加入样本外测试、滚动回测和参数稳定性分析 |
| MVP 范围膨胀 | 延迟交付 | 实盘、高频、优化集群明确排除在 MVP 外 |

## 15. 后续版本方向

- V0.2：参数扫描、策略对比、样本内外切分和滚动回测。
- V0.3：分红送转、配股、退市整理期等公司行为与特殊状态增强。
- V0.4：组合优化、因子研究、横截面选股和归因分析。
- V0.5：模拟盘、券商适配层、实时行情和风险监控。

## 16. 待确认事项

以下问题将直接影响技术架构和 MVP 边界，需在需求评审时确认：

1. 首发使用方式：本地桌面、本地 Web，还是云端多用户系统？
2. 策略频率：仅日线，还是首版必须支持分钟线？
3. 用户画像：零编程用户、Python 用户，还是两者都支持？
4. 数据来源：仅导入文件，还是接入 Tushare、AkShare 或其他指定数据源？
5. 是否将多股票选股和定期调仓作为 MVP 的强制能力？
6. 默认股票池是否排除 ST、退市整理期、上市未满一定天数及流动性不足的股票？
7. 业绩基准默认使用沪深 300，还是允许按策略选择中证全指、上证指数等基准？

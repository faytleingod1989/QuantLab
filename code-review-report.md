## 一、项目整体评估

### 当前状态

这是一个**已通过视觉 QA 的原型（prototype）**，完成了定稿验证（`design-qa.md` 结论: **"final result: passed"**）。前后端链路完整可运行：

```
启动 → 演示数据 → 选择股票池 → 配置均线参数 → 运行回测 → 展示指标与图表
```

代码质量与 PRD 的 MVP 要求之间存在理解差异：**当前"已实现"的是可视化规则引擎 + 有限后端，而非 PRD 定义的标准策略接口 + 完整数据管理平台。**

### 与 PRD 目标的关系

| 维度 | 当前实现定位 |
|------|-------------|
| 用户界面 | ✅ 完成度高，通过设计 QA，截图对比验证通过 |
| 数据管理 | ❌ 仅合成演示数据，无 CSV 导入、日历、复权 |
| 策略接口 | ❌ 仅支持可视化条件组合（均线/RSI），无 Python 代码策略 |
| 回测引擎 | ⚠️ 基本路径可用，但基准伪造、精度不严、规则未版本化 |
| 任务系统 | ❌ 同步阻塞、无队列、无持久化 |
| 可复现性 | ❌ 无数据库、策略快照、数据指纹 |
| 测试 | ❌ 仅 2 个冒烟测试，无单元/金标准测试 |

## 四、严重问题（必须修复）

### [CRIT-01] 基准曲线数据伪造

**文件**: `backend/engine.py:227`

```python
benchmark_growth = np.linspace(0, max(total_return * 0.32, 0.08), len(equity))
```

基准曲线是用 `np.linspace` 线性生成的假数据，且始终锚定到策略收益的 32%。这是**完全伪造的乐观曲线**。所有超额收益、相对比较均无意义。

**影响**: 夏普比率分子使用年化收益（含伪造基准成分不直接但指标对比被污染），用户可能被曲线误导。

**修复**: 使用真实沪深 300 指数数据，或标注"演示数据，非真实基准"。

### [CRIT-02] 无数据持久化

整个系统无任何形式的持久化。回测配置、策略参数、结果在页面刷新后全部丢失。`run.ps1` 重启后一切归零。

**影响**: 与 FR-001"重启后仍可访问"直接冲突，也导致 FR-602（策略快照）无法实现。

**修复**: 引入 SQLite + SQLAlchemy 或 TinyDB（适合本地工具）。

### [CRIT-03] 回测引擎阻塞 API 线程

`engine.py` 的 `run_backtest()` 在 FastAPI 请求处理器中同步执行。特殊风险：

- 多标的 + 长周期场景会阻塞服务器进程（FR-406 要求可取消）
- `data.sample_daily()` 生成数据的开销包含在请求时间内
- 无法跟踪进度（无 WebSocket / SSE）

**修复**: 使用 `BackgroundTasks` + 轮询/WebSocket 进度推送。

### [CRIT-04] 无框架级未来数据防护

PRD 第 9.1 条明确"严禁使用下一交易日数据生成信号"。当前实现仅凭代码规范确保数据窗口，策略代码（若未来支持 Python 策略时）没有任何访问限制。

**修复**: 引入 `DataView` 包装层，在回测循环中按日期切片数据。

## 五、重要问题

### [MAJ-01] RSI 实现与标准 Wilder RSI 不一致

`engine.py:21`:

```python
gain = delta.clip(lower=0).ewm(alpha=1/period, adjust=False).mean()
```

用 EMA 近似替代 Wilder 的 SMMA，结果会系统性地偏移（通常更平滑）。用户在不同平台间对比 RSI 信号会产生歧义。

**影响**: FR-203（技术指标）的验收标准未满足。

### [MAJ-02] 金额精度管理缺失

| 问题 | 代码位置 | 分析 |
|------|----------|------|
| 费用未舍入 | `engine.py:60-63` | `commission = max(...)` 直接使用浮点结果 |
| 持仓均价不展期 | `engine.py:172-173` | 首次买入设置均价，**后续加仓不会更新** `average_cost` |
| 卖出 PnL 使用均价 | `engine.py:124` | `position.average_cost * quantity` — 如果从未加仓则正确，但加仓后均价不会更新 |
| `_round` 精度不对 | `engine.py:66-68` | 4 位小数混用，人民币需要 2 位（分）或更高精度 |

**示例问题** — 多次买入同一标的：
```
第 1 次: 买入 100 股 @10 元 → average_cost = 10
第 2 次: 买⼊ 100 股 @12 元 → average_cost 仍为 10（错误，应为 11）
第 1 次卖出: PnL 基准使用了错误的均价
```

### [MAJ-03] 交易/事件数据被裁剪

`engine.py:253-254`:

```python
"trades": list(reversed(trades[-50:])),
"order_events": events[-20:],
```

前端永远只能看到最近 50 笔交易和 20 条事件。若回测超过此数量，信息静默丢失。

**影响**: 前端"查看全部"按钮无实际功能；与 FR-505 要求支持筛选排序冲突。

### [MAJ-04] 无日志系统

整个后端无 `logging` 模块。回测失败时：

- 无法追踪故障点
- 无法关联任务 ID、策略 ID、数据版本
- `app.py:46` 仅捕获 `ValueError, IndexError`，其他异常会导致 500 且缺乏上下文

**影响**: 违反 PRD 10.3"日志需包含任务 ID、策略 ID、数据版本和异常上下文"。

### [MAJ-05] 多标的现金分配算法有误

`engine.py:151`:

```python
allocation = cash * request.max_position / max(len(active_buys), 1)
```

将全部现金的 `max_position` 比例**等分**给每个买入信号。问题是：

1. 已持有某标的时又出买入信号，`cash` 不变但实际应增量加仓
2. 若某标的已接近 `max_position` 限制，不应继续分配
3. 若 10 个标的同时出信号，每个仅得 `10% × 初始资金 / 10`，可能小于 1 手

### [MAJ-06] 卖出时忽略卖出费用后的净值为结算依据

`engine.py:124`:

```python
pnl = value - fees - position.average_cost * quantity
```

PnL 计算的"成本"是 `average_cost * quantity`，但这包含买入时的费用吗？
`average_cost = (value + fees) / quantity` ✅ 包含了买入费用。这里的 PnL 逻辑一致。但问题在于：

**若 `average_cost` 从未被展期**（MAJ-02），多笔加仓后的 PnL 会出错。

### [MAJ-07] CORS 仅限开发端口

`app.py:14`：

```python
allow_origins=["http://127.0.0.1:5173", "http://localhost:5173"]
```

生产构建后 Vite 预览或静态托管将无法连接。需要环境变量配置。

## 九、结论

### 代码定位

这是一个**视觉设计定稿（design QA passed）的交互原型**。前后端链路跑通了从"选择股票→配置策略→运行回测→展示报告"的完整交互流程，UI 像素级对标设计稿。适合用来验证产品概念和交互体验。

### 与 PRD"MVP"的差距

| 维度 | 原型现状 | PRD MVP 要求 | 差距评估 |
|------|----------|-------------|:--------:|
| 数据管理 | 5 只合成数据 | CSV 导入 + 日历 + 复权 + 主数据 | 2-3 周 |
| 策略 | 可视化条件 | Python 基类 + 参数 + 版本 + 指标库 | 2-3 周 |
| 引擎 | 可用但精度不足 | 金标准测试通过 + 所有 A 股约束 | 1-2 周 |
| 任务 | 同步内存 | 异步队列 + 快照 + 复现 | 1-2 周 |
| 报告 | 基础展示 | HTML 导出 + 多任务对比 | 1 周 |
| 测试 | 2 个 | 完整单元测试 + 金标准 | 1-2 周 |
| **总计** | | | **8-13 周** |

### 最终评价

- **UI/UX**: ⭐⭐⭐⭐⭐ 高质量，通过像素级 QA
- **架构**: ⭐⭐ 单一文件无分层，无数据库
- **正确性**: ⭐⭐ 基本逻辑可跑，但精度和边界有缺陷
- **可维护性**: ⭐⭐ 前后端各 1-2 个文件，无测试
- **安全**: ⭐⭐⭐ 本地工具风险可控
- **PRD 匹配度**: ⭐⭐⭐ 39% 需求覆盖

> **一句话总结**: 这是一个跑通了全链路的交互原型，适合做产品演示和交互验证，但不适合作为量化研究的可信工具使用。如果要达到 PRD 定义的 MVP 标准，还需要 8-13 周的重构和开发工作。

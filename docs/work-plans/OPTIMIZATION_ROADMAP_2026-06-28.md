## 一、总体判断

当前程序已经具备核心雏形：

- 本地 Web 应用可运行。
- 支持 A 股日线数据同步、导入和本地仓库沉淀。
- 支持策略研究、可视化策略配置和回测。
- 已接入 AkShare，并新增 efinance、BaoStock 作为免费行情源兜底。
- 已有基础测试、报告导出、启动脚本和 GitHub 代码管理。

下一阶段不建议继续无序堆功能，而应围绕两条主线优化：

1. 数据链路产品化：让全 A 数据同步、补齐、复用、失败重试变稳定。
2. 策略链路产品化：让选股、买入、卖出、仓位风控分层清晰，配置体验更接近真实量化研究流程。

## 二、优化优先级

| 优先级 | 模块 | 优化方向 | 原因 |
|---|---|---|---|
| P0 | 数据中心 | 自动补齐全 A、增量更新、失败重试、本地仓库复用 | 没有稳定数据，策略和回测都不可靠 |
| P0 | 回测任务 | 后台化、进度显示、可取消、错误明细 | 全 A 回测耗时长，不能阻塞前端 |
| P1 | 策略研究 | 拆分选股策略和交易策略 | 当前一次编辑内容太多，策略结构不清晰 |
| P1 | UI 可读性 | 字体、对比度、表单密度、弹窗布局优化 | 当前小字偏小，复杂配置容易疲劳 |
| P1 | 回测报告 | 增加核心指标、持仓分析、标的贡献 | 方便判断策略是否真的有效 |
| P2 | 工程质量 | 文档归档、配置示例、数据库迁移、CI | 降低后续维护成本 |

## 三、阶段计划

### 阶段 1：数据中心稳定化

目标：让“全 A 数据补齐”从手动点击变成可持续执行的后台任务。

任务：

- 将“继续补齐全 A”升级为后台同步任务。
- 支持自动循环补齐，直到达到目标覆盖率或遇到连续失败阈值。
- 显示同步进度：
  - 总标的数
  - 已覆盖标的数
  - 本批新增标的数
  - 失败标的数
  - 当前使用的数据源
  - 当前批次耗时
- 增加失败明细：
  - 股票代码
  - 失败数据源
  - 失败原因
  - 最近失败时间
- 支持只重试失败股票。
- 支持增量更新：
  - 已有历史数据不重复拉取。
  - 只补缺失日期区间。
  - 默认以后每天只补最近交易日。
- 前端数据中心展示 provider 状态：
  - 本地仓库
  - efinance
  - BaoStock
  - AkShare

验收标准：

- 点击一次“自动补齐全 A”后，无需反复手动点击。
- 中途中断后，重新启动仍能从本地仓库继续补。
- 已下载的数据不会因为再次同步而重复生成多个数据集。
- 前端能看到同步进度和失败原因。

### 阶段 2：回测任务后台化

目标：让大规模回测不会卡住页面，并能清楚知道运行状态。

任务：

- 回测改为后台任务队列。
- 前端展示任务状态：
  - 排队中
  - 运行中
  - 已完成
  - 已失败
  - 已取消
- 支持取消正在运行的回测任务。
- 支持查看回测错误明细。
- 支持全 A 回测时分批执行。
- 记录每次回测的运行参数、数据集版本、策略版本。

验收标准：

- 全 A 回测时页面仍可操作。
- 用户能看到当前跑到哪一步。
- 回测失败时能定位是数据问题、策略问题还是引擎问题。

### 阶段 3：策略研究结构重构

目标：把策略配置从“大而全弹窗”改成分层、分步骤、可理解的研究流程。

推荐结构：

```text
策略研究
├─ 选股策略
│  ├─ 股票池
│  ├─ 过滤条件
│  ├─ 排序规则
│  └─ 最大候选数量
├─ 交易策略
│  ├─ 买入条件
│  ├─ 卖出条件
│  ├─ 持仓规则
│  └─ 风控规则
└─ 策略版本
   ├─ 参数快照
   ├─ 回测记录
   └─ 备注说明
```

任务：

- 策略编辑入口拆成：
  - 编辑选股策略
  - 编辑交易策略
- 选股策略只负责“哪些股票进入候选池”。
- 买入策略只负责“候选股什么时候买”。
- 卖出策略只负责“什么时候离场”。
- 仓位策略只负责“买多少、最多几只、如何分配”。
- 内置模板补充：
  - 筹码稳定 + 均线多头选股
  - 蓄势结构 + 箱体突破选股
  - 趋势跟踪 + 筹码稳定买入
  - 蓄势结构 + 放量突破买入
  - 高位出货信号卖出
  - 破位风险控制卖出

验收标准：

- 用户可以分别编辑选股策略和交易策略。
- 策略页面不再一次性展示过多配置。
- 每个策略模板都能从零通过图形化配置还原。

### 阶段 4：A 股交易规则真实性增强

目标：让回测更接近真实 A 股交易。

任务：

- 完善 T+1 规则。
- 停牌日不可交易。
- 涨停不可买入。
- 跌停不可卖出。
- 支持手续费、印花税、滑点。
- 支持 ST 过滤。
- 支持新股上市 N 日内过滤。
- 支持上市日期不足导致的回测跳过说明。
- 支持除权除息复权一致性检查。

验收标准：

- 回测结果不会出现明显不符合 A 股制度的交易。
- 每笔无法成交的信号有明确原因。
- 回测完成提示不再简单显示“未完成”，而是分类展示跳过、失败、不可交易原因。

### 阶段 5：回测报告增强

目标：让报告能够回答“这个策略到底好不好”。

任务：

- 增加核心指标：
  - 总收益率
  - 年化收益率
  - 最大回撤
  - 夏普比率
  - 胜率
  - 盈亏比
  - 平均持仓天数
  - 最大连续亏损
- 增加分析图表：
  - 权益曲线
  - 回撤曲线
  - 月度收益热力图
  - 持仓数量变化
  - 股票收益贡献排行
- 增加交易明细筛选：
  - 按股票
  - 按日期
  - 按盈利/亏损
  - 按卖出原因
- 报告导出增强：
  - HTML 保留交互
  - PDF 适合归档

验收标准：

- 用户无需看原始交易流水，也能判断策略优劣。
- 报告能解释收益来源和主要风险来源。

### 阶段 6：UI 可读性和交互体验优化

目标：降低使用疲劳，让复杂策略配置也清楚、稳定、好操作。

任务：

- 建立基础 UI 规范：
  - 正文字号不低于 14px。
  - 表单 label 不低于 13px。
  - 说明文字不低于 12px，并提高对比度。
  - 重要数字使用更大字号。
- 优化深色主题对比度。
- 策略配置弹窗改成分步骤布局。
- 数据中心、策略研究、回测中心、报告中心改成真正页面导航。
- 表格、卡片、按钮统一间距。
- 长列表增加搜索、筛选和分页。

验收标准：

- 常用页面不需要放大浏览器也能看清。
- 侧边栏点击后进入对应页面，而不是只弹出对话框。
- 策略编辑页面不再出现控件重叠。

### 阶段 7：工程质量和维护体系

目标：让项目越来越大之后仍然可控。

任务：

- 建立文档目录规范：
  - `docs/prd/`
  - `docs/work-plans/`
  - `docs/code-reviews/`
  - `docs/user-guide/`
- 增加 `.env.example`。
- 增加数据库迁移机制。
- 增加启动自检脚本。
- 增加 GitHub Actions：
  - 后端测试
  - 前端测试
  - 构建检查
- 增加开发者操作手册：
  - 如何启动
  - 如何同步数据
  - 如何跑测试
  - 如何新增策略模板

验收标准：

- 新环境可以按文档启动项目。
- 每次提交能自动跑基础测试。
- 后续改数据库结构不需要临时手动修。

## 四、推荐执行顺序

建议按以下顺序推进：

1. 数据中心自动补齐全 A。
2. 数据同步进度和失败重试。
3. 回测任务后台化。
4. 策略研究拆分为选股策略和交易策略。
5. UI 字体和页面导航优化。
6. A 股真实交易规则增强。
7. 回测报告增强。
8. 工程质量和文档体系完善。

## 六、关键风险

| 风险 | 影响 | 应对方式 |
|---|---|---|
| 免费数据源限流或不稳定 | 全 A 数据同步慢或失败 | 多源兜底、本地仓库缓存、失败重试 |
| 全 A 回测性能不足 | 页面卡顿、任务超时 | 后台任务、分批执行、缓存中间结果 |
| 策略配置复杂度上升 | 用户难以理解 | 分模块、分步骤、模板化 |
| 数据口径不一致 | 回测结果失真 | 统一复权、交易日、字段归一化检查 |
| 文档滞后 | 后期维护困难 | 每个阶段完成后同步更新 docs |

## 七、最终目标形态

理想状态下，QuantLab 应该形成这样的使用流程：

1. 打开本地 Web。
2. 数据中心自动检查本地仓库是否缺数据。
3. 一键自动补齐沪深全 A 日线。
4. 在策略研究里选择或配置选股策略。
5. 单独配置买入、卖出和仓位规则。
6. 提交后台回测任务。
7. 查看进度、结果、报告和交易明细。
8. 根据报告复制策略版本，继续迭代。

这条路径跑顺之后，QuantLab 就不只是一个回测 demo，而是一个真正可长期使用的 A 股本地量化研究工作台。
